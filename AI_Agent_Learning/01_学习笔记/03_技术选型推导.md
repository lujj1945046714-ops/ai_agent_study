# 技术选型的苏格拉底式推导

## 核心原则：解决问题 > 堆砌技术

### 用户的技术直觉（非常好的质疑）
- "历史职位数据避免重复，不用 RAG 也可以做吧？用 RAG 是不是复杂了？"
- "不要硬凑技术点，边做边学"

---

## 问题1：RAG 的真正价值在哪里？

### 场景A：存储历史职位数据
- 需求：避免重复抓取同一个职位
- 简单方案：SQLite 数据库存 (job_id, title, company, fetch_time)
- RAG 方案：向量数据库存职位描述 → 语义搜索
- **提问：** 这个场景真的需要"语义搜索"吗？还是简单的 ID 查重就够了？

**答案：** 简单查重够了，RAG 是过度设计 ❌

---

### 场景B：理解"这个岗位真正做什么"
- 需求：JD 写"负责 Agent 系统开发"，我想知道具体做什么
- 问题：单纯问 LLM，它可能编造（幻觉）
- RAG 方案：建立一个知识库
  - 存储：真实的技术博客、小红书经验分享、GitHub 项目 README
  - 当问"Agent 系统开发是什么"时 → 检索相关真实经验 → 基于事实回答

**提问：** 这个场景需要 RAG 吗？如何避免幻觉？

**答案：** 需要！这是 RAG 的真正价值 ✅
- 不是为了用 RAG 而用，而是为了**基于真实经验而非幻觉**

---

### 场景C：找 GitHub 项目
- 需求："公司需要熟悉 Agent 规划" → 找相关项目
- 方案1：直接搜 GitHub API "agent planning"
- 方案2：建立 GitHub 项目知识库（RAG）
  - 预先爬取热门 Agent 项目
  - 存储：README、技术栈、难度评估
  - 检索：根据需求找到最匹配的项目

**提问：** 实时搜索 vs 知识库，哪个更好？

**答案：** 看时间限制
- 一周项目：实时搜索 GitHub API 就够 ✅（简单）
- 长期产品：建知识库更好（但现在过度设计）❌

## 问题2：什么才是"真正的技术深度"？

### 误区：技术深度 = 用了很多技术
- ❌ 用了 RAG + Agent + Computer Use + ... (堆砌)
- ✅ 深入解决一个核心问题

### 你想展示的技术深度（重新审视）
1. **多模态信息融合**
   - 核心问题：JD(结构化) + 小红书(文本) + LLM(生成) → 可靠结论
   - 技术难点：如何判断信息可信度？如何去重？如何处理矛盾？
   - **这是真深度！** ✅

2. **个性化推荐**
   - 核心问题：不是"找最好的项目"，而是"找最适合你的项目"
   - 考虑因素：你的背景(RAG经验)、时间(一周)、目标(找工作)
   - 技术难点：如何建模"适合度"？
   - **这是真深度！** ✅

3. **Token 优化**
   - 核心问题：多次调用 LLM 很贵，如何减少消耗？
   - 可能方案：缓存、批处理、渐进式筛选（先粗筛再精选）
   - **这是真深度！** ✅

### 不是核心的技术
- RAG 存历史职位：❌ 简单数据库就够
- nanoclaw 记忆存储：❌ 如果不是核心需求就别强行加

## 问题3：如何找 GitHub 项目？（你不知道的部分）

### 需求拆解
"公司需要熟悉 Agent 规划" → "找 GitHub 项目"

### 方案推导（苏格拉底式）
**子问题1：** "Agent 规划"这个需求如何转换成 GitHub 搜索关键词？
- 简单：直接搜 "agent planning"
- 更好：理解"规划"的技术同义词
  - planning, reasoning, task decomposition, ReAct, chain-of-thought...
- **如何找到这些同义词？** → 问 LLM！

**子问题2：** 搜到100个项目，如何筛选？
- 简单：按 star 数排序
- 更好：考虑
  - 难度适中（代码量、复杂度）
  - 有文档（README 质量）
  - 活跃度（最近更新时间）
  - 技术栈匹配（用你熟悉的 Python）

**子问题3：** 如何判断"这个项目能体现 Agent 规划能力"？
- 看 README：是否明确说明了规划算法
- 看代码：是否有 planning/reasoning 模块
- **如何自动化？** → LLM 读 README，判断是否匹配需求

### 完整流程
```
岗位需求("Agent 规划")
  → LLM 提取技术关键词 ["planning", "ReAct", ...]
  → GitHub API 搜索
  → LLM 分析每个项目的 README
  → 评分(难度、匹配度、完整度)
  → 推荐 Top 3
```

**这里的技术深度在哪里？**
- 需求 → 关键词的语义扩展
- 项目质量的多维度评估
- 个性化筛选（适合你的背景和时间）
