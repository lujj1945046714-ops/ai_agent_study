# 真实求职流程分析

## 用户描述的完整流程

### 第一阶段：信息收集
1. **找各大公司 AI Agent 岗位 JD**
   - 目的：了解技能要求
   - 问题：每个公司工作内容描述不同，有些不清楚

2. **多源信息补充**
   - 小红书：找在职员工了解真实工作内容
   - 大模型：理解工作的细致内容
   - 目的：理解岗位"真正做什么"

### 第二阶段：个性化筛选
3. **根据个人偏好筛选**
   - 维度：薪资、工作地点
   - 输出：目标公司列表

### 第三阶段：能力匹配
4. **分析技术背景要求**
   - 这些公司需要什么技术栈

5. **项目搜索与改进**
   - GitHub 找相关项目
   - 参考大模型建议做改进
   - 学习项目并放入简历

## 技术深度分析（待推导）

### 关键问题1：多源信息融合
**挑战：** JD、小红书、大模型回答，这三种信息源的结构完全不同
- JD：结构化（公司、职位、要求、薪资）
- 小红书：非结构化（用户分享、评论）
- 大模型：生成式内容（可能有幻觉）

**技术问题：**
- 如何融合这些信息？
- 如何判断信息的可信度？
- 如何去重和矛盾处理？

### 关键问题2：理解"工作真正做什么"
**挑战：** 不是简单关键词匹配，而是语义理解

例如：
- JD 写："负责 Agent 编排和优化"
- 你需要理解：这是写代码？调参数？还是做架构设计？

**技术问题：**
- 如何从描述中提取真实工作内容？
- 如何判断这个工作是否适合自己？

### 关键问题3：需求到项目的映射
**挑战：** "公司需要 XXX 技能" → "GitHub 上哪个项目能体现这个技能"

例如：
- 需求："熟悉 Agent 规划算法"
- 如何找到：相关 GitHub 项目？
- 如何评估：这个项目是否真的能体现这个能力？

**技术问题：**
- 技能描述 → 项目标签/README 的语义匹配
- 项目质量评估（star数？代码质量？）

### 关键问题4：项目改进的价值判断
**挑战：** 不是所有改进都有价值

**技术问题：**
- 如何判断改进方向是否符合岗位要求？
- 如何确保改进有技术深度（而不是改个UI）？

## 技术解法总结（2026-02-23 补充）

| 关键问题 | 解法 | 技术 |
|---------|------|------|
| 多源信息融合 | MVP阶段先只用JD单一来源，避免过度设计 | - |
| 理解"工作真正做什么" | LLM提取core_work[]字段，JSON格式输出 | LLM + JSON mode |
| 需求→项目映射 | LLM扩展技术同义词 → GitHub API → LLM评估README | GitHub API + LLM |
| 改进建议价值判断 | Prompt中明确要求"技术深度方向"，输入完整JD约束 | Prompt Engineering |
