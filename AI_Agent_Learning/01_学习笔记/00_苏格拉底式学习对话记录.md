## 项目方向确定过程

### 第一轮：发现真实痛点

**提问：你日常遇到什么重复性、复杂的问题？**

**回答：** 找信息很复杂和麻烦
- 想找 AI Agent 工作，不知道需要什么能力
- 不知道这些能力如何学习
- 网上信息复杂，不知道是否符合自己情况
- 找项目很难：要符合技术背景、一周能做、有深度、符合趋势
- 找到了还要实验适不适合，浪费时间

**关键洞察：**
> 你自己的痛点就是最好的项目方向！成千上万的应届生都有同样的问题。

### 第二轮：理解真实求职流程

**提问：如果手动完成"找 AI Agent 工作并制定学习计划"，你会怎么做？**

**回答的完整流程：**

1. **信息收集阶段**
   - 找各大公司 AI Agent 岗位 JD
   - 发现问题：每个公司工作内容描述不同，有些不清楚具体做什么
   - 多源补充：
     - 小红书：找在职员工了解真实工作内容
     - 大模型：理解工作的细致内容

2. **个性化筛选阶段**
   - 根据薪资、工作地点筛选
   - 确定目标公司列表

3. **能力匹配阶段**
   - 分析这些公司的技术背景要求
   - 在 GitHub 找相关项目
   - 参考大模型建议做改进
   - 学习项目放入简历

**关键洞察：**
> 这个流程远比简单的"职位匹配"复杂，涉及多源信息融合、语义理解、个性化推荐等深度问题。

### 第三轮：技术选型推导

**核心原则：** 解决问题 > 堆砌技术，不要硬凑技术点

#### 关键技术判断

**1. Computer Use 的应用**
- ✅ 招聘网站抓取职位数据（nanoclaw）
- ✅ 小红书抓取经验分享（nanoclaw）
- 可选：网站 API（如果有的话）

**2. 大模型的应用**
- ✅ 理解职位信息（LLM API）
- ✅ 提取技能要求
- ✅ 生成改进建议（需要好的 prompt）

**3. RAG 的真正价值**

**误区：** 为了用 RAG 而用 RAG

**正确判断标准：**
- ✅ 需要语义相似搜索
- ✅ 需要基于真实文档避免幻觉
- ❌ 简单精确查找 → 用数据库
- ❌ 数据量很小 → 用数据库

**在本项目中：**
- 职位去重：❌ 不需要 RAG（SQLite 就够）
- 理解"工作真正做什么"：✅ 需要 RAG（基于真实经验而非幻觉）
- nanoclaw 记忆存储：❌ 不需要强行加 RAG（混淆了概念）

**结论：** 一周 MVP 可能不需要 RAG，focus 在核心功能

#### GitHub 项目推荐的技术难点

**挑战：** "公司需要熟悉 Agent 规划" → "找 GitHub 项目"

**解决方案推导：**

1. **需求 → 关键词的语义扩展**
   - 简单：直接搜 "agent planning"
   - 更好：理解技术同义词（planning, reasoning, task decomposition, ReAct, CoT）
   - 实现：让 LLM 扩展关键词

2. **项目质量评估**
   - Star 数
   - 难度（代码量、复杂度）
   - 文档质量（README）
   - 活跃度（更新时间）
   - 技术栈匹配（Python）

3. **个性化筛选**
   - 考虑用户背景（RAG 经验）
   - 时间限制（一周）
   - 目标（找工作展示能力）

**完整流程：**
```
岗位需求("Agent 规划")
  → LLM 提取技术关键词
  → GitHub API 搜索
  → LLM 分析 README
  → 多维度评分
  → 推荐 Top 3
```

#### 改进建议的设计

**目标：** 避免浅层建议（如"换个UI"），确保有技术深度且符合岗位需求

**方法：**
- 提示词工程：明确要求技术深度
- 输入公司 JD：确保改进方向符合需求
- 约束条件：一周时间、现有技术栈

## 技术深度体现点

### 确定要展示的技术能力

1. **多模态信息融合** ✅
   - JD（结构化）+ 小红书（非结构化文本）+ LLM（生成式）
   - 技术难点：如何判断可信度？如何去重？如何处理矛盾？

2. **个性化推荐** ✅
   - 不是"找最好的项目"，而是"找最适合你的项目"
   - 技术难点：如何建模"适配度"？

3. **Token 优化** ✅
   - 多次 LLM 调用的成本控制
   - 方案：缓存、批处理、渐进式筛选（粗筛 → 精选）

### 技术指标设计

1. **召回率**：系统找到的合适职位 / 真实存在的合适职位
2. **准确率**：系统推荐的职位中真正合适的比例
3. **成功率**：Agent 多步骤任务完整执行成功的比例
4. **鲁棒性**：在不同网站、页面结构下的适应能力
5. **Token 消耗优化**：完成同样任务的 token 数量对比

**验证方法：**
- 准备测试集（50个真实职位）
- 人工标注匹配结果
- 系统运行并对比
- 量化指标展示

## 苏格拉底式提问法的体会

**什么是苏格拉底式提问？**
- 不直接给答案
- 通过一系列问题引导思考
- 让学习者自己推导出结论
- 深入理解"为什么"而非"是什么"

**本次对话中的应用：**
1. 不直接说"做XX项目"，而是问"你遇到什么问题？"
2. 不直接给架构图，而是问"你手动做会分几步？"
3. 不说"用XX技术"，而是问"这个场景真的需要吗？"
4. 持续挑战假设，避免为了技术而技术

**收获：**
- 对项目方向有了真实认同（因为是自己的痛点）
- 理解了每个技术选择的"为什么"
- 学会了判断技术是否必要的方法
- 建立了"解决问题优先"的思维
